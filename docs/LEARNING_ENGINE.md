# Learning Engine

## Назначение

Learning engine отвечает за проверку ответов, расчет прогресса, выбор следующего упражнения и интервальное повторение. Он должен быть детерминированным, тестируемым и независимым от Vue.

## Входы и выходы

Входы:

- content package;
- текущее persisted state;
- действие пользователя: ответ, подсказка, пропуск, импорт прогресса;
- фиксированное время, переданное параметром, а не прочитанное внутри функции.

Выходы:

- результат проверки;
- обновленный progress state;
- следующая рекомендация;
- события для UI: feedback, mastery change, review scheduled.

## Результат проверки

```ts
type CheckResult = {
  status: 'correct' | 'partially-correct' | 'incorrect'
  checkKind: CheckSpec['kind']
  matchedRules: string[]
  failedRules: string[]
  feedback: string
  normalizedAnswer?: string
}
```

`feedback` объясняет проверяемое правило, но не притворяется выводом compiler/runtime.

## Pipeline проверки

1. Получить `Exercise` и `CheckSpec`.
2. Нормализовать ввод, если стратегия это допускает.
3. Для Kotlin-like ответа выполнить простую токенизацию без попытки стать compiler.
4. Применить check strategy.
5. Вернуть `CheckResult` с явным `checkKind`.
6. Записать attempt через store action.

## Нормализация

Нормализация может:

- унифицировать окончания строк;
- обрезать внешние пробелы;
- схлопывать повторяющиеся пробелы там, где это разрешено;
- нормализовать пробелы вокруг операторов;
- сохранять строки и символы как значимые токены.

Нормализация не должна менять семантически важные части ответа.

## Токенизация

Токенизатор нужен для учебных проверок, а не для полного парсинга Kotlin. Он различает:

- identifiers;
- keywords;
- literals;
- operators;
- punctuation;
- comments;
- whitespace, если стратегия делает его значимым.

Если задача требует полноценной семантики Kotlin, ее нужно переформулировать как bounded exercise.

## Mastery

Для каждого `skillId` хранится:

```ts
type SkillMastery = {
  skillId: string
  score: number
  correctWithoutHints: number
  correctWithHints: number
  debugCorrect: number
  transferCorrect: number
  reviewCorrect: number
  lastPracticedAt: string
  nextReviewAt: string
  intervalDays: number
  ease: number
}
```

Тема считается освоенной только если выполнены все условия:

- `score` выше порога;
- есть несколько правильных ответов без подсказок;
- выполнены разные типы упражнений;
- есть успешное исправление ошибки;
- есть успешный transfer;
- есть успешное отложенное повторение.

## Обновление mastery

Начальные значения:

- `score = 0`;
- `intervalDays = 1`;
- `ease = 2.3`.

Правила:

- правильный ответ без подсказок повышает score сильнее всего;
- правильный ответ с подсказками повышает score ограниченно;
- частично правильный ответ дает небольшой прирост только при совпадении ключевых правил;
- неправильный ответ снижает score и сокращает interval;
- debug, transfer и review учитываются отдельными счетчиками.

Формулы должны быть простыми, явными и покрытыми unit-тестами после подключения Vitest.

## Интервальное повторение

Детерминированная схема:

```text
quality 5: interval = round(previousInterval * ease)
quality 4: interval = max(previousInterval + 1, round(previousInterval * 1.5))
quality 3: interval = 1
quality 1-2: interval = 0
```

`ease` изменяется в ограниченном диапазоне `1.3..2.8`. Карточки с `interval = 0` возвращаются в текущую сессию, карточки с `interval > 0` попадают в очередь повторения.

## Выбор следующего задания

Приоритет:

1. Просроченные повторения.
2. Непройденные обязательные упражнения текущей темы.
3. Упражнения того же навыка другого типа.
4. Transfer-задание.
5. Следующая доступная тема по графу зависимостей.

Алгоритм не использует случайность. Если нужен эффект разнообразия, используй stable ordering по `exerciseId` и состоянию прогресса.

## Импорт и экспорт прогресса

Экспорт:

- сериализует только пользовательское состояние;
- включает `schemaVersion`, `contentVersion`, timestamp и checksum;
- не включает учебный контент.

Импорт:

- валидирует JSON;
- проверяет версию схемы;
- запускает миграции;
- показывает конфликт contentVersion;
- применяет данные только после успешной проверки.

## Тестируемость

Pure domain modules должны тестироваться без браузера. Обязательные наборы тестов после подключения Vitest:

- нормализация;
- токенизация;
- каждая check strategy;
- mastery update;
- review scheduling;
- migrations;
- import/export validation.
