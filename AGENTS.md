# AGENTS.md

Этот репозиторий - клиентское образовательное SPA для глубокого и быстрого изучения Kotlin. Документ задает правила для всех последующих изменений в проекте.

## Назначение проекта

- Приложение работает полностью на клиенте: без собственного backend, регистрации, облачной синхронизации, платных API, AI-интеграций, токенов и внешнего сервиса выполнения Kotlin.
- Прогресс хранится локально через Pinia PersistedState и `localStorage`; обязательны экспорт и импорт прогресса.
- После первого открытия приложение должно поддерживать офлайн-работу для уже поставляемого контента.
- Kotlin-код пользователя проверяется только детерминированными правилами: нормализация, токенизация, сравнение ожидаемых токенов, структурные правила, обязательные и запрещенные конструкции, известные допустимые решения и заранее определенные результаты.
- Интерфейс обязан честно показывать тип проверки. Не добавляй кнопку `Run`, если Kotlin реально не компилируется и не выполняется.

## Текущее состояние репозитория

- Стек: TypeScript, Vite, Vue 3, Composition API, `<script setup lang="ts">`, Pinia, Pinia PersistedState, Tailwind CSS 4, Reka UI, Lucide Icons.
- `src/main.ts` создает Vue-приложение и подключает Pinia PersistedState.
- `src/App.vue` пока пустой; не заполняй его большим монолитом.
- Роутер, тестовый раннер, PWA, поиск и контентная система еще не настроены.

## Команды

- Установка: `npm ci` для воспроизводимой установки по `package-lock.json`; `npm install` только при осознанном обновлении зависимостей.
- Разработка: `npm run dev`.
- Typecheck: `npx vue-tsc -b`.
- Сборка: `npm run build` - запускает `vue-tsc -b` и `vite build`.
- Preview production-сборки: `npm run preview`.
- Линтинг: отдельный `lint`-script пока не настроен. Если добавляешь ESLint, зафиксируй команду здесь и в [docs/TESTING.md](docs/TESTING.md).
- Unit/e2e тесты: scripts пока не настроены. Если добавляешь Vitest или Playwright, обнови [docs/TESTING.md](docs/TESTING.md).

## Архитектурные ограничения

- Разделяй UI, доменную логику, учебный контент, проверяющие функции и хранилище.
- Страницы и компоненты не должны содержать бизнес-логику обучения, алгоритмы проверки ответов или миграции состояния.
- Контент должен быть версионирован и валидируем. Изменение контента не должно ломать сохраненный прогресс.
- Состояние пользователя хранится через типизированные Pinia stores; формат persisted state имеет `schemaVersion` и миграции.
- Основные функции не зависят от сети. Сетевые запросы допустимы только для статических assets, поставляемых самим приложением.
- Крупные разделы контента и тяжелые редакторы загружай лениво.
- Проверяющие функции должны быть чистыми и покрываться тестами, когда тестовый раннер будет подключен.

## TypeScript и Vue

- Используй строгую типизацию и текущие compiler options: `noUnusedLocals`, `noUnusedParameters`, `erasableSyntaxOnly`, `noFallthroughCasesInSwitch`.
- Не используй `any` без короткого документированного обоснования рядом с кодом. Предпочитай `unknown`, discriminated unions и явные типы.
- Для Vue используй Composition API и `<script setup lang="ts">`.
- Держи route-level страницы тонкими: загрузка данных, композиция layout и делегирование в feature/domain модули.
- Для UI используй существующий стек: Tailwind CSS, Reka UI для доступных primitives, Lucide Icons для иконок.

## Правила работы с контентом

- Не создавай фиктивный учебный контент ради заполнения экрана.
- Каждая тема должна иметь учебный цикл: краткое объяснение, разобранный пример, сборка кода из частей, заполнение пропусков, написание без подсказки, исправление ошибки, перенос знания в новый контекст, интервальное повторение.
- Тема считается освоенной только после нескольких успешных ответов без подсказок, разных типов заданий, исправления ошибочного кода, задания на перенос и отложенного повторения.
- Каждое упражнение указывает тип проверки: exact, normalized, token, structural, construct-rule, known-solution или bounded-result.

## Запрет на заглушки

- Не добавляй десятки пустых компонентов, страниц или stores.
- Не заменяй реализацию списком TODO.
- Если функциональность еще не готова, не показывай ее пользователю как работающую.
- Не объявляй этап завершенным, если доступные проверки не запускались.

## Обязательная проверка перед завершением задачи

Минимум для текущего состояния проекта: `npm run build`.

Если в ходе задачи добавлены scripts или зависимости для качества, также запускай соответствующие проверки: lint, typecheck, unit tests, e2e tests. Если проверку невозможно запустить, укажи точную причину в финальном отчете.
